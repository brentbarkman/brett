# Daily Briefing — Design Spec

## Overview

Implement the daily briefing feature end-to-end. The briefing auto-generates on first app open of the day, shows a static fallback when AI isn't configured, uses proper timezone handling, and has comprehensive tests and evals.

Rename from "morning briefing" to "daily briefing" throughout.

## Data Model

### User.timezone

Add `timezone` field to the User model:

```prisma
timezone      String  @default("America/Los_Angeles")
timezoneAuto  Boolean @default(true)
```

- IANA timezone string (e.g., `America/New_York`, `Europe/London`)
- `timezoneAuto`: when true, client auto-updates the stored timezone on each app open
- When false, the stored value is used regardless of device timezone

### Timezone Helper

New function in `@brett/business`:

```typescript
function getUserDayBounds(timezone: string, now?: Date): { startOfDay: Date; endOfDay: Date }
```

Returns UTC `Date` objects representing the start and end of "today" in the user's timezone. Used by the assembler, API routes, and the summary endpoint.

## API Routes

### `GET /brett/briefing` (existing, fix timezone)

Returns today's cached briefing or null. Fix: use `getUserDayBounds(user.timezone)` instead of local `new Date()` math.

Response: `{ briefing: { sessionId: string, content: string, generatedAt: string } | null }`

### `POST /brett/briefing/generate` (existing, fix timezone)

Streaming SSE generation. Fix: pass user's `timezone` into the assembler input so it computes correct day bounds and formats event times in the user's local timezone.

The route handler must query `user.timezone` from the database before constructing the input, since the auth middleware user object does not include timezone fields. Use `prisma.user.findUnique({ where: { id: user.id }, select: { timezone: true } })`.

Input to assembler: `{ type: "briefing", userId: string, timezone: string }`

### `GET /brett/briefing/summary` (new, no AI required)

Lightweight endpoint returning raw counts. No LLM involved.

Response:
```typescript
{
  overdueTasks: number;
  dueTodayTasks: number;
  todayEvents: number;
  overdueItems: Array<{ title: string; dueDate: string }>;
}
```

Uses `getUserDayBounds(user.timezone)` for date calculations. `overdueItems` returns up to 3 overdue task titles and due dates.

### `PATCH /users/timezone` (new, added to existing users router)

Body: `{ timezone: string, auto: boolean }`

Validates timezone string against `Intl.supportedValuesOf('timeZone')`. Updates `User.timezone` and `User.timezoneAuto`.

### `GET /users/me` (existing, extend response)

Add `timezone` and `timezoneAuto` to the response so the client can check current values.

## Assembler Changes

### `BriefingContext` type update

```typescript
type BriefingContext = {
  type: "briefing";
  userId: string;
  timezone: string;  // new
}
```

### `assembleBriefing` changes

- Receives `timezone` from input
- Uses `getUserDayBounds(input.timezone)` for all date queries (overdue, due today, events)
- Replaces `currentDateLine()` with timezone-aware date: `new Date().toLocaleDateString("en-CA", { timeZone: input.timezone })` so the LLM sees the correct "today" for the user
- Formats calendar event times using `toLocaleTimeString("en-US", { hour: "numeric", minute: "2-digit", timeZone: input.timezone })`
- Updates the user message from "morning briefing" to "daily briefing"
- `BRIEFING_SYSTEM_PROMPT` opening line changes from "morning briefing" to "daily briefing" (export name stays the same)
- Adds timezone context to the system prompt: `Current timezone: ${input.timezone}`

## Client

### `useTimezoneSync` hook (new)

- On app startup, reads `Intl.DateTimeFormat().resolvedOptions().timeZone`
- Fetches user profile to check stored timezone and `timezoneAuto` flag
- If `timezoneAuto` is true and detected timezone differs from stored, sends `PATCH /users/timezone`
- Runs once per app session via a ref guard

### `useBriefing` hook (modified)

- On mount, if `hasAI` and no cached briefing exists for today, auto-trigger `regenerate()` once
- `hasAutoGeneratedRef` prevents double-triggers
- The client does NOT compute "today" — it trusts the server's timezone-aware response (`null` means no briefing for user's today)
- No other behavioral changes

### `useBriefingSummary` hook (new)

- Calls `GET /brett/briefing/summary`
- Always enabled (no AI required)
- Returns raw counts for the static fallback card

## UI

### Rename

`MorningBriefing` → `DailyBriefing` (component name, file name, all imports/exports).

### Props interface

```typescript
interface DailyBriefingProps {
  content: string | null;
  isGenerating?: boolean;
  summary?: {
    overdueTasks: number;
    dueTodayTasks: number;
    todayEvents: number;
    overdueItems: Array<{ title: string; dueDate: string }>;
  } | null;
  hasAI: boolean;
  onDismiss: () => void;
  onRegenerate?: () => void;
}
```

### TodayView rendering

Remove the `briefing.hasAI` gate from the render condition. The `DailyBriefing` component always renders — it decides internally which state to show based on `hasAI`, `content`, and `summary` props.

```tsx
{isBriefingVisible && (
  <DailyBriefing
    content={briefing.content}
    isGenerating={briefing.isGenerating}
    summary={summary.data}
    hasAI={briefing.hasAI}
    onDismiss={() => setIsBriefingVisible(false)}
    onRegenerate={briefing.regenerate}
  />
)}
```

### Three states

**1. AI briefing** (has AI + content):
- Header: "Daily Briefing" with refresh button and dismiss
- Bullet-point content from LLM
- Subtle "generated at" timestamp
- Streaming state shows content filling in as text arrives

**2. Generating** (has AI + no content yet):
- Same card chrome
- "Generating your briefing..." with animated indicator
- Content streams in as it arrives

**3. Static fallback** (no AI configured):
- Same card chrome, no refresh button
- Structured counts: "3 tasks due today · 2 overdue · 1 meeting"
- Up to 3 overdue task titles as bullet points if any exist
- Subtle CTA: "Configure AI in Settings for a personalized briefing"
- If truly empty: "Your day is clear — no tasks or meetings."

### Dismiss behavior

- Dismiss hides for current session (state in TodayView)
- Reappears on next TodayView mount (next app open)

## Settings

New "Timezone" section in Settings:
- Display current timezone
- Toggle: "Use device timezone" (controls `timezoneAuto`)
- When toggle is off: dropdown of IANA timezone strings for manual override

## Testing

### Unit: Assembler (`packages/ai/src/context/__tests__/assembler.test.ts`)

- `assembleBriefing` with timezone — correct day bounds for `America/New_York`, `Asia/Tokyo`, `Pacific/Auckland`
- Timezone boundary: user in UTC+13 at 11pm → correct "today" in UTC terms
- Empty data: no tasks, no events → "No tasks due and no calendar events today."
- All overdue, nothing today → only overdue section
- Mixed data → all sections present in correct order
- Event time formatting in user's timezone

### Unit: Timezone helper (`packages/business/src/__tests__/timezone.test.ts`)

- `getUserDayBounds` correctness for various timezones
- DST transitions: spring forward and fall back
- Extreme offsets: UTC+14 (Kiribati), UTC-12 (Baker Island)

### API Integration (`apps/api/src/__tests__/briefing.test.ts`)

- `GET /briefing` returns null when no briefing exists
- `GET /briefing` returns today's cached briefing, not yesterday's
- `GET /briefing` timezone boundary: briefing created at 11pm UTC (today in UTC) but tomorrow in `Asia/Tokyo` should not return for a Tokyo user
- `GET /briefing/summary` returns correct counts with seeded data
- `PATCH /users/timezone` validates IANA strings, rejects invalid input
- Auth required on all routes

### LLM Evals (`evals/fixtures/briefing-quality.json`)

Eval cases that feed known data through the assembler and validate LLM output:

- **Format compliance**: 3-5 bullet points, under 120 words
- **No hallucination**: only references tasks/events from input data
- **Empty handling**: acknowledges light day without inventing tasks
- **Specificity**: uses actual names, times, attendees — not vague
- **Overdue emphasis**: overdue tasks first, mentions how late they are

Each fixture: input data + assertion criteria. Eval runner: assembler → LLM → validate against criteria.

## Migration / Deploy Order

1. Apply Prisma migration (adds `timezone`, `timezoneAuto` to User) — must land before API code changes
2. Deploy API with updated routes and assembler
3. Deploy desktop with new hooks and renamed component

The schema default (`"America/Los_Angeles"`, `timezoneAuto: true`) means existing users get a sensible fallback. On their next app open, the client auto-detects and updates their timezone.

## Out of Scope

- Re-enabling the `morning_briefing` skill in the omnibar (can be done later as routing sugar)
- Embeddings / Layer C memory for briefing personalization
- Push notifications for briefing readiness
- Briefing scheduling (cron-based pre-generation)
